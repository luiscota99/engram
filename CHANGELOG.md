# Changelog

All notable changes to Engram will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Fixed

- **Dual FTS write path removed (schema v23) — the standing corruption generator found by the July 17 audit.** The v6 triggers and `index_in_fts()` both indexed items, with mismatched `item_id` types (trigger: INTEGER, code: TEXT), invisible to each other's cleanup DELETEs. Every add created a degraded duplicate row (no tags, no vector), and the update triggers deleted the *good* text row on any core-table UPDATE — so a routine usage bump could silently destroy an item's semantic searchability (verified live: mistakes #23–25 were duplicated within hours of being added). v23 drops every `fts_*` trigger — `index_in_fts()` is the sole owner (fresh DBs never had the triggers; all insert paths verified to call it) — and repairs existing drift in place: duplicate integer rows deleted, orphan integer rows re-keyed to TEXT. `doctor`'s `integrity_report` now checks the failure shapes directly (duplicate `(item_type, id)` groups, non-TEXT ids) rather than only aggregate counts, and the daily self-check files a high-severity alert if a second writer ever returns.

### Added

- **Retrieval feedback (schema v22) — `engram feedback`, MCP `memory_feedback`: reward and discourage in ranking, never in existence.** Ranking had reward machinery (log usage boost) but no working feedback loop: `usage_count` almost never moved (54 of 58 skills at zero), and `record_usage(success=False)` silently ignored the failure — a memory that failed you got boosted anyway. Now `engram feedback <type:id> --helped|--unhelpful` (agents: `memory_feedback`, ids come from the `[TYPE #id]` recall banners) records explicit events in one shared `retrieval_feedback` table; rank-time aggregation is a single batch query across all types. Demotion is deliberately stronger than reward (precision over recall, same stance as the recall/guard gates). Two invariants set by the user: **non-use is not a signal** — dormant knowledge scores exactly as before, because you can't leverage everything all the time — and **deletion is never automatic**: the daily self-check files net-negative items (≤ −2) as inbox *decisions* proposing archival, and only the user approves. `record_usage(success=False)` now routes to negative feedback instead of bumping the count.

- **Crash-proof session checkpoints (schema v21) — `engram hook checkpoint`, `engram resume`, MCP `memory_resume`.** Closes the resume gap observed live: a session died mid-flight on a spend limit, and reconstructing "where did I leave off?" from a fresh session cost ~100k tokens of transcript archaeology — the exact problem Engram exists to prevent. A Claude Code `Stop` hook now upserts one row per (project, session) after **every** agent turn: the last user prompt, the agent's final reply (the natural handoff), and the git HEAD at that moment. Because it fires per turn — not at session end — it survives spend-limit cutoffs, crashes, and closed laptops; a tool-only turn never blanks a previously captured handoff, and API-error banners/subagent sidechains are filtered out. `engram resume` (or `memory_resume`) then answers in one call: latest checkpoint, handoff text, and any commits made since. Checkpoints are operational state, **not** memories — never FTS-indexed or embedded, never surfaced by search. `engram bootstrap` installs the Stop hook alongside recall and guard.
- **Recall relevance gate — injecting nothing beats injecting noise.** Hybrid search always returns a nearest neighbor, so the auto-recall hook was injecting unrelated memories on conversational turns (observed live: a TCP-server skill surfaced for "help me grant access"). `engram hook recall` now requires real lexical overlap between the prompt and each hit — same precision-over-recall stance the guard already proved — filtering per hit and injecting nothing when no hit qualifies. Purely-semantic paraphrase matches are the accepted cost.
- **Relationship questions the user decides — `not_related` edges + `engram unlink`.** Some links between memories are spurious or one-directional (an MTG note referencing a database pattern doesn't make them related). The daily self-check now detects **cross-domain links** (endpoints whose tag domains don't overlap) and files each as an inbox *decision*: keep it (do nothing) or reject it (`engram unlink a b related && engram link a b not_related`). The answer is durable knowledge: `not_related` is a first-class negative edge that suppresses the question forever (in either direction) and records the distinction itself as memory. Untagged or same-domain links are never questioned — no nagging without signal.
- **Per-agent "mini brains" — `engram brain new|list|path`.** A brain is a self-contained, isolated memory under `~/.engram/brains/<name>`: its own DB, config, telemetry, and a generated launcher that scopes every `engram` command to it (pins `ENGRAM_DB_PATH`). The point is isolation — a specialized agent's recall and guard fire on ITS domain, not on a global pool of unrelated projects (the cross-contamination that dilutes reuse). `engram brain new <name>` scaffolds it and prints how to point an agent's Claude Code hooks at that brain; `list` shows each brain's memory counts; `path` prints its DB path for scripting. Generalizes the hand-rolled `./bin/brain` pattern into a first-class command — the substrate for turning one agent into a trained specialist.
- **Typed relationships between memories (schema v20) — `engram link`, `engram relations`, MCP `memory_link`.** The genuinely useful, LLM-era slice of a knowledge graph without the OWL/RDF/SPARQL weight: a small **closed** vocabulary of directional edges (`supersedes`, `refines`, `causes`, `contradicts`, `depends_on`, `related`) over the memories Engram already stores. The LLM is the reasoner; these just let recall follow the graph — `memory_read_item` now attaches an item's relationships. Kept deliberately low-friction (the lesson of memory nobody reuses): one call to link, and supersession edges are recorded **automatically** when a merge/consolidation invalidates an item. No ontology authoring, no reasoner, no query language.
- **Pre-action guard + pre-commit guard (enforcement levels 3–4) — `engram hook guard`, `engram guard`.** Complements auto-recall with a cautionary check: before an Edit/Write/Bash (a Claude Code `PreToolUse` hook) or before a commit (a `pre-commit` hook running `engram guard --staged`), Engram surfaces **known mistakes/patterns** relevant to the change. Precision over recall: unlike recall, the guard requires real **lexical overlap** (shared terms), so an unrelated action never triggers a false "you're repeating a mistake" warning. Warn-only by default (never blocks); `--strict` turns it into a confirmation (hook) or a non-zero exit (commit). `engram bootstrap` installs the PreToolUse hook alongside auto-recall. Note: the guard reads the developer's **local** Engram DB — CI has none, so enforcement lives at the developer's machine, not in CI.

- **Auto-recall enforcement hook — `engram hook recall`, generated by `engram bootstrap`.** Recall was advisory (rules told the agent to search); nothing enforced it, so reuse sat near zero. This makes recall automatic: a Claude Code `UserPromptSubmit` hook runs `engram hook recall`, which searches memory for the incoming prompt and injects the top matches as context — whether or not the agent chooses to search. `engram bootstrap` writes the hook into the project's `.claude/settings.json` (idempotent, preserves existing settings). Two invariants: injected text is framed as **reference data, not instructions** (it enters context automatically and must not be trusted as commands), and surfacing is logged (`audit source=hook`) but does **not** bump `usage_count` — "surfaced" is not "used", so the reuse metric stays honest.

- **ROI / help measurement — `engram roi`, `engram audit on|off|status`, MCP `memory_roi`.** Closes the loop that made "how much has Engram helped?" unanswerable: the search-audit log existed but was off by default and no report consumed it. `engram audit on` persistently enables auditing (stored in `~/.engram/config.json` next to the DB; an explicit `ENGRAM_AUDIT_LOG` still wins; local-only — query + top-5 result ids). `engram roi` reads that log plus realized reuse (usage counts, capture→reuse) and the reflex-rung token floor, and prints an **honest one-line verdict** — including "auditing is OFF, can't measure" or "capture-heavy, reuse-light" when that's the truth. No invented numbers.

- **Skill validation (schema v19, ADR 0008)** — proof that a memory changes behavior, not just that it's stored (Superpowers' "TDD for skills" rigor applied to personal memory). A `skill_tests` scenario+assertion is run through the LLM cold (baseline) and with the memory injected (treatment): only **validated** = baseline-fails + treatment-passes counts — a test that passes cold proves nothing. Verdicts: validated / redundant / ineffective / regressed / untested (graceful without an LLM; `contains` grading is deterministic). `engram validate add|run`; `engram route` badges proven prior art `✓validated`; the daily self-check flags reused-but-unvalidated skills. Reuse says "retrieved a lot"; validation says "demonstrably helps".

- **`read_only` safety flag on reflexes (schema v18)** — a dimension orthogonal to `kind`. Mutating is the safe default (a script never earns free-run treatment by inference — only by explicit `engram reflex approve --read-only`). Read-only reflexes invoked by an agent over MCP run freely; **mutating reflexes now require an elicitation confirmation per use** (approval grants the capability, the confirmation authorizes this use — the two-tier trust model), closing a real gap where `handle_reflex_call` executed any approved reflex with no per-use gate. MCP tool descriptions are tagged `[read-only]` / `[mutating — asks for confirmation]` so agents know before calling.

- **Inbox + monitors + self-check (schema v17)** — the asynchronous half of the approval principle: agents and monitors *propose*, the user decides. `inbox` table (alerts + decision requests, severity-ordered, `finding_key` dedup); `engram inbox` / `engram decide <id> --approve [--run]` is the only proposal→execution path. Reflex `kind=monitor`: firing files a deduped alert instead of demoting. `ENGRAM_CHANGE` journal makes every reported mutation revertible-by-information. `engram schedule` writes marker-managed crontab entries (the OS owns the clock — no daemon). `engram self-check` (daily) files promotion candidates, flaky reflexes (<70% over 5+ runs), placeholder mistakes, pending embeddings, consolidation clusters and reuse warnings as decisions. Delivery via the user-approved `notify` reflex (`engram notify-init`, osascript default, `ENGRAM_NOTIFY_MIN_SEVERITY` threshold). MCP: `memory_propose_decision`; `memory_route` warns on open high/critical items before system changes.

- **Heavy review sweep (2026-07-07, "true recall" edition).** Recall: routing search now retrieves wide (30 candidates) and decides narrow — both channels capped at 5 BEFORE fusion, which dropped correct skills at candidate generation; the router also matches tasks directly against approved reflex names/descriptions. FTS rebuilt with **porter stemming** (schema v16, rowids preserved — no re-embed): "committing" now matches "commit". Recency no longer halves a cold memory's relevance (floored at 0.75); RRF weight raised 15→50 so fusion actually reorders; per-type `rank_multiplier` (defined since the registry existed, never read) is finally wired. Measured: seeded R@1 82%→85%, MRR 0.899→0.917, R@5 stays 100%; the field-observed routing miss now resolves to its reflex.
- **Token diet.** MCP tool list 6,669→5,138 tokens/session (legacy add_* tools removed from the list — handlers still accept them; conversation/prompt folded into unified `memory_add`; `auto_extract` merged into `suggest_capture`; boilerplate trimmed). Search responses drop the duplicated injection-policy preamble (~85 tok/call); `memory_route` uses a one-line wrapper (~90 tok/call); `read_item` returns compact JSON.
- **Persistent query-embedding cache** (`~/.engram/embed_cache.db`, L2 under the in-process LRU, `ENGRAM_EMBED_CACHE=off` to disable): repeated CLI searches 1.02s→0.10s — the CLI is a fresh process per call and previously always paid the ~100ms Ollama embed.
- **Reflex drafting correctness**: `bash -n`/`py_compile` gate at approval (broken scripts are refused instead of burning runs), `set -euo pipefail` mandated (silent mid-script failures recorded false 'ok' runs and corrupted the reuse metric), params schema inferred from the LLM draft (agents see real arguments in the MCP tool), and related mistakes injected into the draft prompt as preflight guards. Reflex output now clips head+tail — failures live at the END of logs and head-truncation discarded exactly that signal.

### Fixed

- **A dead scheduler was invisible — the monitor couldn't detect its own death.** The daily `self-check` was scheduled in cron but had been crashing at startup for months (on macOS, cron/launchd can't read a repo/venv under `~/Desktop` without Full Disk Access), so nothing was actually being monitored — and nothing said so. Added a heartbeat (`record_self_check_heartbeat`) and `scheduler_status()`; `run_self_check` now files a **high-severity inbox alert** when a self-check is scheduled but hasn't run (or the install sits under a macOS-protected folder), and `engram schedule` warns about a TCC-protected path at creation time. A blind monitor is now a detected, surfaced condition.
- **The scheduled self-check watched too narrow a signal — structural drift rotted silently.** `run_self_check` only alerted on `embedding_status` rows explicitly marked pending/stale, so the problems that actually degrade a DB — FTS↔core drift, FTS↔vector drift, `failed` embeddings never retried, orphaned status/tag rows — were invisible to the monitor (only `engram doctor` caught them, and doctor isn't scheduled). Added a structured, side-effect-free `doctor.integrity_report()` and wired it into `run_self_check`, which now files those integrity findings to the inbox (idempotent by finding key). The monitor now watches what actually breaks.
- **Cold-start of a CPU-only Ollama could fail a whole reembed batch.** Loading an embedding model takes several seconds (~8s for `nomic-embed-text` on CPU); when that latency landed on the first request of a batch it blew the per-request timeout, and two such failures tripped the 20s dead-host cooldown — so the rest of the sweep was marked `failed` without ever being tried, leaving residual vector drift. Added `embeddings.warm_up()` (a long-timeout priming call that also clears any stale cooldown) and call it at the start of `reembed_stale()`; `embed_text()` now accepts an explicit `timeout`. Captured as a mistake in the reference DB.
- **Migration output corrupted the MCP JSON-RPC stream**: `database.py`/`migrations.py` printed progress to stdout, which the MCP server uses for framing — the first tool call after a schema upgrade broke the session. Routed to logging.
- **Dedup fingerprints diverged between CLI and MCP** (MCP omitted `prevention` for mistakes): shared canonical content builders in `memory_ops`.
- `git rev-parse` was forked on every search for project affinity — memoized per path.
- `urllib.request` import deferred (~30ms off every non-embedding CLI command).

### Added

- **Reflex run history (schema v15)** — every execution recorded in `reflex_runs` (status, duration); `engram efficiency` reports measured success rates, not just streaks. Reflex timeout raised to 300s for real workflows (test suites, builds).
- **`ENGRAM_EMBED_MAX_CHARS`** — optional per-document embedding length cap; embedding time scales ~linearly with input length, so this is the honest bulk-ingest speed lever. Measured on Apple Silicon: embeddinggemma is 5x *slower* than nomic-embed-text via Ollama despite fewer params (noted in the model table) — benchmark before switching models.
- **Action Ladder** — the framework surface for token efficiency + action correctness (ADR 0006). `engram route` / MCP `memory_route` answers "cheapest correct way to do this task" in one budgeted call: an approved reflex to invoke, prior art to follow, or reason-then-capture; known mistakes surface as pitfall warnings on every rung. Reflexes auto-demote after 2 consecutive failures (approval revoked, failure captured as a mistake — schema v14 adds `fail_streak`); skills used 5+ times surface as promotion candidates in `engram health`; `engram efficiency` reports measured ladder stats with conservative token floors only.
- **Reflexes (experimental)** — proven skills can be promoted to executable, human-approved scripts that agents invoke as MCP tools (`reflex_<name>`) instead of re-reasoning through the workflow text each call (~50 tokens vs thousands, deterministic). Lifecycle: `engram promote <skill_id>` drafts a script (LLM-assisted or template), a human reviews and runs `engram reflex approve <id>` (approval pins the script hash — later edits un-approve it), and approved reflexes appear in `tools/list` across every MCP client. Params are passed as `PARAM_<KEY>` env vars, never interpolated into the script; unapproved or tampered scripts refuse to run. Schema v13 adds the `reflexes` table.
- `engram bootstrap --omit-project-integration` and sentinel file `.omit-agent-integration` (project root) to skip Cursor rules + Antigravity `instructions.md` while still initializing the database and MCP when applicable.
- README guidance on explicitly asking for retrieval (Agent Integration).
- Hybrid search merges semantic vs lexical rankings with reciprocal rank fusion (RRF); FTS query tokenization aligns with BM25/ranking tokenizer; tests in `tests/test_rrf.py`.
- In-process LRU cache for query embeddings (`src/embeddings.py`, 256 entries) — the Ollama HTTP round-trip dominated search latency (~85% in profiling); repeated queries in a long-lived MCP session now skip it. `clear_embedding_cache()` resets it.
- Schema parity regression test (`tests/test_migrations.py::test_fresh_schema_matches_migrated_schema`) — fails if `SCHEMA_SQL` and the migration chain diverge.

- **Pluggable embedding backends** — `ENGRAM_EMBED_URL` now selects an OpenAI-compatible `/v1/embeddings` endpoint (with `ENGRAM_EMBED_API_KEY`); unset keeps local Ollama; `disabled` turns embeddings off cleanly (short-circuits before any HTTP).
- **Flexible embedding dimensions** — `engram migrate-embeddings --target-model <model>` probes the model's real output dimension and rebuilds `vec_memory` at the new width when it differs (vectors are regenerated from FTS content). Write paths validate against the live `schema_meta.vec_dimension` instead of a hardcoded 768.
- **`src/config.py`** — every environment variable Engram reads, documented and resolved in one module.
- **MCP elicitation (spec 2025-06-18+)** — destructive tools (`memory_gc` non-dry-run, `memory_sleep`) ask the user for confirmation via `elicitation/create` when the client advertises the capability; clients without it keep the previous behavior.
- **Tokens-per-query in benchmarks** — `longmemeval_bench` and `engram_retrieval_bench` report `context_tokens` per query and `avg_context_tokens` in the aggregate (the 2026 memory-benchmark comparison axis alongside accuracy).
- **Claude Code skill** — `engram claude-skill` installs `claude-skills/engram-memory/SKILL.md` into `~/.claude/skills/` so Claude Code searches memory before non-trivial tasks and captures lessons afterward.
- **`engram install`** — one-shot setup: detects Cursor / Claude Code / Antigravity on the machine and wires all global integrations (Cursor MCP config, Claude skill, `~/.gemini/AGENTS.md` snippet); `--all` forces every integration.
- **`engram import-claude-memories`** — imports Claude Code's file-based memories (`~/.claude/**/memory/*.md`) into Engram, idempotent by content hash, so native Claude memories become searchable across every tool.
- **Capture→reuse metric** — `engram health` (and the MCP health tool) now report what share of memories captured 30+ days ago were ever used again, per type and overall, with a recommendation when reuse is low. This is the capture-quality signal.
- **Real LongMemEval retrieval eval** — `longmemeval_bench.py --oracle-file longmemeval_oracle.json` ingests all haystack sessions into one corpus (~950 sessions) and grades session-level Recall@k/MRR against `answer_session_ids`, per question type, with tokens-per-query. First full run (2026-07-06): session-level R@5 0.538 / MRR 0.442 over 940 sessions, 218 ms and 374 tokens per query, fully local; results and caveats published in `benchmarks/BENCHMARKS.md`.
- **Leaner MCP search injection** — `memory_search` default limit 10→5 with rank-aware snippets (500 chars for the top hit, 150 below): ~34% fewer context tokens per search on a real corpus (845→560) while the top hit carries 3× more detail; `estimate_context_tokens` in benchmarks now mirrors this injected format instead of raw row sizes.
- Connection reuse on the search hot path — one connection per search (was 8), plus a `connection_scope` helper for multi-step operations.
- sqlite-vec pinned to >=0.1.9 (proper DELETE/space reclamation in vec0 tables).
- **Batched + deferred embeddings** — `embed_batch` (one HTTP round-trip per 16 docs with per-item fallback), `ENGRAM_DEFER_EMBED=1` write-fast mode (FTS rows land instantly, vectors fill in via batched `engram reembed`), used by the LongMemEval ingest; corpus ingestion is now crash-resumable.
- **Explicit-date ranking boost** — queries mentioning a date ("in May 2023", "2024-03") boost items whose date matches by ISO prefix. A directional heuristic ("first"/"most recent") was benchmarked on LongMemEval and removed (no R@5 effect, slightly negative MRR) — see benchmarks/BENCHMARKS.md.
- **Turn-window chunked ingestion** — `create_conversation_chunked` indexes overlapping turn-windows as sibling rows so buried single-sentence evidence gets its own vector; `--chunked` flag on the oracle benchmark. Benchmarked 2026-07-08: **regression** (aggregate R@5 0.540→0.500, preference 0.367→0.200) — windows crowd out the top-k without parent-level aggregation; stays opt-in and off by default, diagnosis in BENCHMARKS.md.
- **Reuse-aware capture** — `suggest_capture` consults per-type reuse rates (`get_reuse_rates`) and warns when a suggested type historically never gets retrieved again.

### Fixed

- **Embedding normalization (schema v12)** — Ollama's legacy `/api/embeddings` returns unnormalized vectors while the newer `/api/embed` returns unit vectors; mixing them under euclidean KNN silently partitioned the vector index (queries only matched same-endpoint documents). All vectors are now L2-normalized in code, and migration v12 rescales existing stored vectors in place — no re-embedding needed.
- Fresh databases were missing schema v11 objects (`memory_facts` table, `superseded_by` columns, `codebase_knowledge.file_mtime`) because they were added only in migrations, never in the baseline `SCHEMA_SQL`. `memory_invalidate` with a reason crashed on new installs.
- `engram suggest-consolidate` and `run_sleep` crashed (or reported `clusters_found == 2` on an empty database) — both treated the `(clusters, skip_reason)` tuple from `find_consolidation_candidates` as the cluster list.
- Flaky search performance test: the 500ms budget included a live Ollama embedding call (800ms+ on a cold model load); the query embedding is now stubbed so the test measures the SQL/ranking path it was written to guard.
- `engram doctor --repair` built the orphaned-tags DELETE by string interpolation; now parameterized.
- Silent `except: pass` handlers in `temporal.py`, `maintenance.py`, and `database.py` now log (one of them had already hidden the fresh-install schema bug).

## [1.1.0] - 2026-04-21

### Added
- Unified `memory_add` MCP tool — consolidates `memory_add_mistake`, `memory_add_pattern`, and `memory_add_skill` into a single dispatching tool with explicit `type` field
- `memory_suggest_capture` MCP tool — heuristic-driven auto-drafting of memory entries from task description and outcome; no LLM required
- `engram browse` CLI command — interactive curses-based TUI for browsing and searching memory entries without leaving the terminal
- `engram suggest-capture` CLI command — surface capture suggestions directly from the CLI
- Modular CLI architecture: `src/cli/` package with 7 focused command modules (`memory`, `codebase`, `sync`, `bootstrap`, `maintenance`, `session`, `tools`)
- `src/capture.py` — standalone heuristic capture engine with domain inference, signal detection, and structured draft generation
- `src/browse.py` — curses TUI browser with search, type-filter, and detail view
- Three-mode Cursor rule system: **LIGHT** (default, one search at session start), **FULL** (auto-escalates on complexity signals), **COMMITTEE** (explicit opt-in only via `@engram committee`)
- `SearchResults` list subclass in `src/search.py` to safely carry a `semantic_status` attribute on search result lists
- `phases` and `phase_requirements` columns added to the `workflows` table (schema + migration v8)
- Comprehensive test suite: `tests/test_capture.py` (23 tests), `tests/test_cli_commands.py` (12 tests), `tests/test_workflow.py` (16 tests)
- **Retrieval benchmark hardening (v1.1.0):** `benchmarks/grading.py` with id-based ground truth (`expected_type` + `expected_item_id`) and title fallback; `benchmarks/test_queries.json` aligned to the seed; optional `--failure-detail` and `failed_query_ids` + `top_hits_detail` in JSON on `engram_retrieval_bench.py`
- `evals/` — optional labeled real-DB evaluation (`evals/README.md`, `real_queries.json.example`); not run in CI by default
- `benchmarks/inject_noise.py` — reproducible distractor stress test; interpretation notes in `benchmarks/BENCHMARKS.md`

### Fixed
- MCP schema mismatch: `memory_add_skill` tool definition used `trigger_desc` instead of `trigger` — corrected to match the handler
- `advance_phase` bug: phase advancement previously ignored custom workflows stored in the `workflows` table; now correctly loads phases and role requirements via `_load_workflow_phases`
- `init_session_state` read-before-commit bug: calling `get_session_state` inside the same write transaction made the inserted row invisible; moved the read to after the write connection commits
- `SearchResults` attribute error: `search()` tried to set `.semantic_status` on a plain `list`, causing `AttributeError` when semantic search was skipped — fixed with `SearchResults(list)` subclass
- N+1 query in `get_project_affinities`: replaced per-result queries with a single batched `WHERE (item_type, item_id, project_id) IN (VALUES ...)` query
- Wrong GitHub URL in the bootstrap command
- MCP codebase tools (`memory_index_file`, `memory_query_codebase`, `memory_get_stale_files`) no longer spawn subprocess calls — reimplemented with direct Python database calls

### Changed
- CLI entry point updated from `src.cli_legacy` to `src.cli:main` (the new modular package)
- Adaptive Cursor rule simplified from 122 lines to ~50 lines with clearer mode definitions
- Committee workflow is now strictly opt-in: removed automatic escalation triggers; only activates on explicit `@engram committee` / `use committee` / `committee workflow` phrases
- `cli.py` renamed to `cli_legacy.py` (deprecated, kept for reference only during transition)
