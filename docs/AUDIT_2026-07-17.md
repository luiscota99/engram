# Engram Audit — July 17, 2026

**Method:** two parallel audit agents (post-June-modules code audit; data-layer/scaling audit with read-only queries against a copy of the real DB) + facilitator verification of the critical finding against the live DB.
**Scope guard:** the June 2026 RAGE audit's findings (CLI/MCP duplication, table_map drift, god-modules, untested MCP, missing benchmarks) were excluded; most have since been addressed (`memory_ops`, `item_registry`, benchmark gate, LongMemEval run, MCP handler tests).

## Critical

### 1. Dual FTS write path is an active corruption generator (VERIFIED LIVE)

The v6 triggers and `index_in_fts()` both index items, with a type mismatch that hides them from each other:

- Triggers (`src/migrations.py:96-206`) insert `NEW.id` as INTEGER into `memory_fts.item_id`; `index_in_fts()` (`src/database.py:614`) inserts TEXT and its cleanup DELETE binds text — never matching the trigger's integer row.
- **Every add via `memory_ops` creates two FTS rows**: the good one (tags + vector + embedding_status) and a trigger duplicate (no tags, no vector). Verified live: mistakes #23–25 (added 2026-07-17) each have an integer-typed, tag-less duplicate; 149 FTS rows vs 146 core items; 3 duplicate groups.
- **The destructive half:** `after_update` triggers DELETE `WHERE item_id = CAST(OLD.id AS TEXT)` — matching the *good* row — then reinsert a bare integer row. `record_usage()` UPDATEs the core table on every use, so routine usage bumps destroy an item's embedded/tagged FTS row and orphan its `vec_memory` row. (Verified: 0 items degraded so far — the last `doctor --repair` reset the board; the 3 dups are post-repair.)

**Fix:** one owner for FTS writes. Either drop the v6 triggers (code path owns indexing — recommended, since only `index_in_fts` maintains vec + embedding_status), or make triggers the sole indexer with consistent types. Then a one-time dedup repair. Everything else in retrieval — ranking, evals, learned weights — sits on top of this; fix it first.

## High

### 2. Drift detection is aggregate-only; repair is nuclear
`doctor.py` compares total counts (offsetting drift reads as "perfect"); any mismatch triggers `rebuild_fts()` — full delete + re-embed of everything (multi-hour at 100k items, and re-embedding is exactly the cold-start cascade of Mistake #16). Fix: per-item diff (core LEFT JOIN fts on type + CAST(id)), repair only the delta.

### 3. Backups can silently miss data; no restore path exists
`backup_before_migration()` copies only the main DB file — under WAL, recent commits live in `-wal` and are missed. Use the SQLite backup API or `VACUUM INTO`. JSON export omits 11 tables (sessions, transcripts, tags, item_tags, pins, relations, facts, skill_tests, inbox, checkpoints, item_projects). No `restore` function exists anywhere — restore is untested by construction.

### 4. Monitor notifications run a subprocess while holding the DB write lock
`reflex.py:386-407` + `inbox.py:82`: when a monitor reflex files an alert, `_maybe_notify()` runs the notify reflex subprocess inside the caller's open write transaction. A hung notify channel holds the write lock (up to 300s); the notify reflex's own bookkeeping INSERT then blocks against the outer uncommitted transaction in the same process → guaranteed `OperationalError` after ~5s, swallowed at `logger.warning`. Fix: collect pending notifications, dispatch after commit.

### 5. Every search full-scans the FTS table for pinned items
`get_pinned_items` joins `memory_fts` on non-MATCH predicates — FTS5 can't index that; `EXPLAIN QUERY PLAN` confirms `SCAN VIRTUAL TABLE`. Same pattern in `get_recent` and dedup's Jaccard fallback (Python-side Jaccard over every row of a type per add). Fine at 150 items; a wall at 100k. Fix: denormalize pins into a small snapshot table; use FTS MATCH for dedup candidate selection. (vec0 KNN is brute-force too: ~300MB scanned per query at 100k×768 — partition/quantize when volume demands.)

## Medium

### 6. `decide --approve --run` marks items `executed` even when the run failed (`inbox.py:165-173`) — set `executed` only on `outcome["ok"]`.
### 7. Mutating-reflex confirmation gate fails open on elicitation errors (`reflex.py:553-562`) — distinguish "client doesn't support elicitation" (proceed) from "confirmation requested but round-trip failed" (refuse).
### 8. Check-then-act races: inbox approve/run can execute twice; finding_key dedup can double-file; `ensure_tag`/`get_or_create_project` are SELECT-then-INSERT (`IntegrityError` under concurrent hooks). Fix: guarded UPDATE (`WHERE status='open'`, check rowcount), `INSERT ... ON CONFLICT DO NOTHING`.
### 9. Unlocked auto-migration race: concurrent hooks can all enter `run_migrations`; v16's FTS swap interleaved across processes can drop the table. Fix: `BEGIN IMMEDIATE` + version re-check. Also: no explicit `busy_timeout` set anywhere; deferred transactions make read-then-write fail with un-retried `SQLITE_BUSY_SNAPSHOT`.
### 10. No retention anywhere: `checkpoints`, `reflex_runs`, `retrieval_feedback`, `session_transcripts`, decided `inbox` rows, and `audit.jsonl` grow forever; `summarize_audit_log` re-reads the whole JSONL per ROI call. Fix: age/size-based pruning in the daily self-check; audit-log rotation.
### 11. Guard hook embeds unbounded payloads: `_guard_query_from_tool_input` falls back to `str(tool_input)` uncapped (`hooks.py:168`) — clip to ~400 chars.
### 12. Reflex flaky-verdict uses all-time history — a repaired reflex is re-flagged forever; window to last N runs.
### 13. Embedding failures have no automatic retry; `migrate_embeddings_to_model` processes one 50-item batch and returns.

## Low

- Self-check sub-scans swallow failures at DEBUG while the heartbeat stamps success (`maintenance.py` — the "blind monitor" failure mode one level down). Surface per-section errors.
- Relations traversal is N+1 per edge (`relations.py:153-195`) — batch like `feedback_totals`.
- Checkpoint prompt filter drops genuine prompts starting with `<` — match actual injected markers instead.
- Brain launcher breaks from non-repo CWD in dev installs and on interpreter paths with spaces.
- `delete_item` leaves orphans in six soft-FK side tables; FK enforcement is per-connection only.
- Trigger-written FTS rows also have narrower content (no `prevention`, empty tags) — compounds finding 1.
- Downgrades untested, no migration dry-run; `_test_migration_v2` ships in prod DBs.
- `embed_cache.db` (14MB for 839 rows) never VACUUMs.

## Data-quality invariant suite (dbt-style asserts for self-check/CI)

- No duplicate `(item_type, CAST(item_id AS TEXT))` in `memory_fts`; FTS↔core bijection per type.
- FTS↔`vec_memory`↔`embedding_status` rowid bijection; `typeof(item_id)='text'` uniformly.
- Unit-norm stored vectors; `embedding_status.embedding_model` = current model for `ready` rows.
- No soft-FK orphans in `item_projects`, `item_pins`, `skill_tests`, `memory_relations`, `retrieval_feedback`, `memory_facts`.

## What's already solid

- Batching discipline (usage counts, feedback totals, affinities: single VALUES/IN queries).
- Reflex trust model: params as env vars (no shell interpolation), SHA-256 approval pinning, parse-check before approval, demotion feeds recall.
- Invariants live in code: reference-data banners, surfaced≠used, precision-over-recall gates, blank-never-overwrites checkpoint upsert.
- Embedding-model machinery: per-item state machine, dimension guard, L2-norm defusal of the endpoint-mixing trap.
- Ingestion idempotency (seed/import/upsert paths safe to re-run); WAL on; up-migrations regression-tested including fresh-vs-migrated schema equality.

## Strategic directions (asked: data-engineering tooling? gradient descent/LLM-style learning? other?)

### Data engineering: adopt the practices, not the stacks
At engram's volumes (MBs, not TBs), dbt/Airflow/DuckDB would be dead weight against the local-first bet. What *is* worth stealing: **declarative data-quality invariants** (the suite above, run by self-check daily and CI on the seeded DB), **retention policies** as first-class config, **WAL-correct backups with a tested restore path**, and **single-owner write paths** (finding 1 is precisely what happens without one). That's the data-engineering discipline; SQLite + self-check remains the right engine.

### Learned ranking: yes — and the prerequisites landed this week
`ranking.py` has ~20 hand-tuned constants (`RECENCY_HALF_LIFE_DAYS=90`, `AFFINITY_BOOSTS`, `RRF_WEIGHT=50` "was 15 — cosmetic", intent multipliers, feedback weights…). Nobody knows if they're right; they were set by feel. The project now has everything needed to *fit* them instead:

1. **Now (days, biggest win-per-effort): offline weight fitting.** The eval harness already computes R@5/MRR programmatically over labeled query sets (`evals/real_queries.json`, the 100-query benchmark, LongMemEval oracle). Treat the constants as a parameter vector, optimize with coordinate ascent / random search / CMA-ES (gradient-free "descent" — the scorer isn't differentiable and doesn't need to be), train/holdout split across queries to avoid overfitting, CI-gate the fitted config. This is learning-to-rank without a single new dependency.
2. **Next (weeks, once data accumulates): a learned linear reranker.** `retrieval_feedback` + the now-enabled audit log accumulate (query, surfaced, helped/unhelpful) triples — exactly LTR training data. A logistic regression over the existing features (BM25, cosine, recency, usage, affinity, feedback) stays explainable, local, dependency-free, and can be re-fit weekly by a reflex. Gate on eval improvement before adopting each re-fit.
3. **Later (highest ceiling, highest cost): embedding fine-tuning.** Contrastive fine-tune of the local embed model on engram's own corpus pairs. Wait until 1–2 are exhausted; the identified symbolic lever (query-date-aware ranking for the temporal-reasoning 0.45 / preference 0.30 LongMemEval gaps) is cheaper and should come first.
4. **Optional inference-time lever: local LLM as reranker** of the top-20 for high-stakes recalls (route/resume), trading latency for precision. Engram already has pluggable LLM plumbing (`llm.py`); keep it opt-in.

### The other way: close the loops that already exist
The most engram-native improvement isn't new ML — it's that the system now *measures itself* (audit on, feedback events, ROI report, skill validation harness) but doesn't yet *act* on those measurements automatically. A weekly reflex that: re-fits ranking weights on fresh eval+feedback data → runs the benchmark gate → proposes the new config via inbox (user approves) would make engram self-tuning under the same "propose, don't decide" principle that governs everything else. That's gradient descent in spirit, applied at the system level — and it's a story no vendor memory currently tells.

## Recommended order

1. Kill the corruption generator (finding 1) + one-time dedup repair + invariant suite so it can never return silently.
2. Concurrency + backup correctness (findings 3, 4, 8, 9) — they're all small, and hooks made engram genuinely multi-process this week.
3. Offline weight fitting against the eval suite (strategy #1).
4. Retention policies (finding 10) before the logs get big enough to matter.
5. Learned reranker once feedback volume justifies it (strategy #2).
